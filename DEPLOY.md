# Deploying KakouCalc to Cloudflare Pages

The site is a static Astro build. There is no server and no runtime API call —
all reference data is baked in at build time.

## 1. Cloudflare Pages setup

| Setting | Value |
| --- | --- |
| Build command | `npm run build` |
| Build output directory | `dist` |
| Node version | 20 or later (`NODE_VERSION` env var) |

`npm run build` runs `scripts/fetch-data.mjs` first (refreshing FX/CPI), then
`astro build`. Use `npm run build:offline` to skip the data fetch.

## 2. Environment variables

| Variable | Required | Purpose |
| --- | --- | --- |
| `CF_PAGES_URL` | automatic | Cloudflare injects this. `astro.config.mjs` reads it so canonical/og:url/sitemap resolve to the deployed host with nothing to configure. |
| `SITE_URL` | once you have a custom domain | Overrides `CF_PAGES_URL`. Set to e.g. `https://kakoucalc.com` so canonical points at the real domain, not the `.pages.dev` one. |
| `FRED_API_KEY` | optional | Enables the CPI refresh. Free key: <https://fred.stlouisfed.org/docs/api/api_key.html>. Without it the committed CPI table is used unchanged. |

**Set `SITE_URL` as soon as a custom domain exists.** If both the `.pages.dev`
host and the custom domain serve the site, canonical tags are what stop them
being treated as duplicate content.

## 3. Daily data refresh

FX rates move constantly; CPI updates monthly. Rebuild daily to pick them up:

1. Pages → your project → **Settings → Builds & deployments → Deploy hooks**
2. Create a hook (e.g. `daily-data`). Copy the URL.
3. Trigger it once a day. Either:
   - **Cloudflare Worker Cron Trigger** — a Worker with
     `crons = ["0 5 * * *"]` that does `fetch(HOOK_URL, { method: "POST" })`, or
   - any external scheduler that can POST to a URL.

### Why the data is fetched at build time, not in the browser

- **Quota.** ExchangeRate-API's free tier is ~1,500 requests/month (~50/day).
  Fetching per visitor exhausts it at trivial traffic — it would break exactly
  when the site starts working. One build/day is flat regardless of traffic.
- **Key exposure.** FRED requires an API key. A key in client-side JS is public
  to anyone who opens devtools. Here it never leaves the build environment.
- **Reliability.** No runtime network dependency, no CORS, no loading state, and
  no third-party outage taking the calculators down.

If a fetch fails the build still succeeds and the previous values are kept —
`src/data/reference.json` is committed, and `src/lib/calculators.js` falls back to
its built-in table if that file is absent. Stale-but-labelled beats broken.

## 4. What is deliberately NOT in the markup

- **`FAQPage` schema** — Google removed FAQ rich results on 2026-05-07 and
  dropped the reporting in June 2026. The markup is still valid but renders no
  snippet, so it is not worth adding.
- **`SoftwareApplication` schema** — its rich result requires an
  `aggregateRating` or review. KakouCalc has neither. Inventing ratings is a
  fabricated record and a manual-action risk, so it is omitted. Add it honestly
  if real reviews ever exist.

`BreadcrumbList` (per calculator) and `WebSite` **are** included — breadcrumbs are
a supported rich result and only became expressible once each calculator had a
real URL.

## 5. Realistic example defaults (done)

Every calculator now ships a genuine worked example in its static HTML instead of
`$0.00`. Realistic inputs live in `src/data/examples.js` (one set per calculator),
and `src/pages/calc/[id].astro` uses them to pre-fill the form AND prerender a real
computed result. The `scripts/verify-math.mjs` "example defaults" block asserts
every one renders a valid, finite, non-error figure, so a field change can't
silently reintroduce a blank. The eight calculators that also have a guide
(`src/data/guides.js`) reuse the guide's exact scenario, so the pre-filled tool
demonstrates the very example the guide explains.

This is an Astro-only presentation concern; it lived entirely in
`src/pages/calc/[id].astro`, never in the retired SPA.

## 6. E-E-A-T pages (done)

`src/pages/methodology.astro` and `src/pages/about.astro` carry the trust signals
money-topic pages need: how the math is built and independently verified, cited
data sources, review dates, and editorial identity. A per-calculator review byline
(linking to /methodology) and an `Organization` publisher schema are in place.
`src/data/site.js` holds the shared identity constants — `contactEmail` is set to
the owner's real address (their choice, 2026-07-23; swap for a domain address in
that one line when one exists). **`reviewer` remains a placeholder — no author or
credential is fabricated; fill it only with a real person.**

## 6b. Social card, internal linking & blog (done)

- **`og:image`** — a real 1200×630 card lives at `public/og-image.png`, generated by
  `scripts/gen-og-image.mjs` (re-run it if branding changes). `Base.astro` emits
  `og:image` + `twitter:image` as absolute URLs on every page.
- **Related calculators on every page** — `calc/[id].astro` shows curated links on
  guided calculators and same-category siblings on the rest, so no page is a dead end.
- **Blog** — `src/data/articles.js` + `/blog` (listing) and `/blog/<slug>` (articles,
  `BlogPosting` schema). Top-of-funnel explainers that funnel into calculators; 3 done,
  same depth-first / reviewed discipline as the guides. Expand via `articles.js`.

## 6c. Explanatory content on every calculator (thin-content fix)

Every calculator ships written, indexable explanatory text below the form — the fix
for the "tool-only thin inventory" an AdSense/SEO audit flags when the form and
result are the only markup. Two layers, both rendered by `src/pages/calc/[id].astro`:

- **Generated, on all 77** (`src/lib/explain.js`): a "How to use" section built from
  the calculator's real field list, and a "Worked example" narrated from the
  engine's own computed result on the pre-filled example. Nothing is invented — both
  are derived from the tool's own data, so the prose cannot drift from the tool.
- **Hand-written depth, in reviewed batches** (`src/data/explainers.js`): an accurate
  "The formula", "Key terms" and "FAQ" for the calculators reviewed so far (first
  batch: 18, dated by `EXPLAINERS_REVIEWED`). Each formula is written to match the
  engine's own `compute()`. A calculator with no entry still gets the two generated
  sections; the 8 calculators with a `guides.js` entry keep their fuller guide (it
  overrides the generated layer entirely).

Guarded by `verify-math.mjs` §8: every calculator must render a substantial,
non-throwing explainer, and each authored entry must be well-formed and not
duplicate a guide. Extend coverage by adding entries to `explainers.js` in reviewed
batches — same YMYL, accuracy-first discipline as the guides.

## 6d. Trust & policy pages (done)

The AdSense-mandatory trust pages exist and are linked from the footer on every
page (`Base.astro`): `/privacy`, `/terms`, `/contact` (built 2026-07-23), joining
the existing `/about` and `/methodology`. Written to be TRUE of this site — no
boilerplate claims of data we don't collect or features we don't have:

- **/privacy** — inputs never leave the browser, no accounts/analytics/own cookies,
  localStorage theme only; honest third-party disclosures (Cloudflare hosting,
  Google Fonts) plus the Google-required AdSense cookie text with the Ads Settings
  and aboutads.info opt-out links and an EEA/UK consent note.
- **/terms** — educational-only/no-advice, "as is" accuracy limits, liability,
  content license, the embed-widget license (attribution links must stay), and
  acceptable use. No invented governing-law clause.
- **/contact** — the real `SITE.contactEmail` (mailto; no form by design — the
  site collects no data), with routing for error reports (tied to the
  verification suite), suggestions, embedding, and privacy questions.

If practices ever change (analytics added, a form, etc.), update /privacy and its
effective date in the same commit.

## 6e. Review polish (done 2026-07-23)

- **Favicon** — `public/favicon.svg` (the brand calculator glyph in the primary
  purple), linked from `Base.astro`. SVG-only; add a PNG/ICO fallback if legacy
  Safari matters later.
- **Custom 404** — `src/pages/404.astro` (friendly copy + popular-calculator links
  resolved via CALC_INDEX). Astro emits `dist/404.html`, which Cloudflare Pages
  serves automatically for unknown paths.

## 8. Result charts & embeddable widgets (done)

**Charts** (`src/lib/charts.js`, Astro-only): a dependency-free inline-SVG line/area
chart under the result for a curated set (`CHARTS`, currently 10 calculators) — an
amortization curve (loan balance, read from the prerendered table) and growth curves
(compound, 401k, HSA, mutual-fund fees, CD, retirement planner, retirement
calculator, business forecast, inflation). Growth series are generated by
**re-running the calculator's own `compute()` across its time axis**, so the chart is
always the engine's own numbers and can't drift; `verify-math.mjs` §7 guards them.
Axes start at zero (no misleading truncation). Extend by adding a `CHARTS` entry.

**Embeddable widgets** — the backlink engine. `/embed/<id>` (`src/pages/embed/[id].astro`)
is a compact, standalone, iframe-able version of each calculator: `noindex` +
canonical to `/calc/<id>`, and excluded from the sitemap (`astro.config.mjs` filter).
The "Embed this calculator" box on each calculator page hands out a copy-paste snippet
whose visible `<a>` tags link back to KakouCalc — those anchors on the host page are
the actual backlinks that build domain authority. Framing works because Cloudflare
Pages sends no `X-Frame-Options` by default; **do not add a DENY/SAMEORIGIN frame
header** or the widgets stop embedding.

## 9. The old files (removed)

`app.js`, `index.html` and `styles.css` at the repo root were the pre-migration
SPA. That SPA routed every calculator through a `#/calc/<id>` hash fragment, which
search engines (and the AdSense crawler) do not index as separate pages — so the
whole site read as one thin, dynamic-form page. They have been **deleted**; the
build never used them, and leaving a hash-routed SPA in the tree is exactly the
signal an SEO/AdSense audit flags.

What replaced them, all driven by the same data:

- `src/pages/index.astro` — the homepage, linking to every calculator with a real
  crawlable `<a href="/calc/…">` (no fragments).
- `src/pages/calc/[id].astro` — one real, prerendered URL per calculator
  (`/calc/<id>`), with a worked example, guide, and internal links baked into the
  static HTML.
- `src/lib/calculators.js` — the engine (extracted verbatim, DOM-free).
- `src/styles/global.css` — the stylesheet.

Because the SPA is gone, `scripts/verify-math.mjs` no longer runs its old
"cross-app sync" check (it compared `app.js` against `calculators.js`); the engine
is now single-source. The pre-deletion copies are not tracked here — recover them
from your own backup/version history if ever needed.
